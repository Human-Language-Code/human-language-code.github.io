üèóÔ∏è Core Concepts

1 Mandatory Two-Object Structure

Every HLC specification must start with exactly two top-level objects in an array:

Entry-Level Object

Defines the project's structure, framework, and global rules. Sets up the foundation for how the entire project should be organized and implemented.

Top-Level Object

Specifies the type of software being built (e.g., mobile app, web service, calculator). Defines the main functionality and purpose.

[
  {
    /* Entry-Level Object - Project Structure */
    "name": "DJANGO-STYLE",
    "type": "entry_level"
  },
  {
    /* Top-Level Object - Software Type */
    "name": "CALCULATOR-APP",
    "type": "top_level"
  }
]

2 Required Object Fields

Every HLC object must contain these three essential fields:

N

name

A unique identifier for the object (e.g., "DJANGO-STYLE", "USER_AUTH", "PAYMENT_GATEWAY")

T

type

The role of the object (entry_level, top_level, feature, control, directive, literal)

P

prompt

A natural-language explanation of what the object does or represents

Optional Field:

L

linkers

An array of child objects that provide additional detail or refinement to the parent object

üß© Object Types

EL

Entry-Level

type: "entry_level"

Sets up project-wide structure, framework choices, and global rules. Always appears first in the array.

Example: Django project structure, React app setup, microservices architecture

TL

Top-Level

type: "top_level"

Defines the main software type and primary functionality. Always appears second in the array.

Example: Calculator app, blog platform, e-commerce site, game engine

F

Feature

type: "feature"

Describes specific functionality or capabilities. Linked to parent objects through linkers.

Example: User authentication, file upload, search functionality, payment processing

C

Control

type: "control"

Dictates behavior, timing, or operational parameters. Controls how features work.

Example: Auto-save every 5 seconds, rate limiting, cache expiration, retry logic

D

Directive

type: "directive"

Enforces rules, standards, or requirements. Often nested under other objects via linkers.

Example: Use HTTPS only, follow REST API standards, implement GDPR compliance

L

Literal

type: "literal"

Provides fixed values or concrete specifications. Cannot have linkers (terminal nodes).

Example: Version 4.2, port 8080, database name "users_db", API key format

üå≥ Structure & Linking

Understanding Linkers

Linkers are the powerful mechanism that connects parent objects to their children, creating the hierarchical structure that makes HLC incredibly expressive and organized.

Linker Structure Components

N
name

The role or relationship type

D
description

How the child relates to parent

V
value

The actual child object

"linkers": [
  {
    "name": "framework",
    "description": "Project framework specification",
    "value": {
      "name": "DJANGO",
      "type": "directive",
      "prompt": "Use Django framework with latest version"
    }
  }
]

        "name": "interface",
        "description": "User interface specification",
        "value": {
          "name": "RESPONSIVE-UI",

      }
    ]

‚≠ê Best Practices

1 Start Simple, Build Complexity Gradually

‚úÖ Recommended Approach

  • Begin with basic Entry-Level and Top-Level objects
  • Add features incrementally through linkers
  • Use descriptive, meaningful names and prompts
  • Keep initial specifications focused and clear

‚ùå Common Pitfalls

  • Creating deeply nested hierarchies from the start
  • Using vague or generic object names
  • Skipping the mandatory two-object structure
  • Over-complicating initial specifications

2 Naming Conventions & Standards

Object Names

Use UPPERCASE with hyphens for clarity

DJANGO-STYLE USER-AUTH PAYMENT-GATEWAY

Linker Names

Use lowercase descriptive terms

framework authentication database

Prompts

Clear, actionable sentences

"Use Django framework" "Implement OAuth2"
